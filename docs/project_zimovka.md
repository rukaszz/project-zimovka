# Project Zimovka

## 概要

Zimovkaは越冬を意味するロシア語，ズィモーフカ．
共産主義国家的な雰囲気をベースにした舞台設定である．
ただあくまでもベースであり，要素として重視するのは「冬・革命期・工業化・古い帝国・鬱屈とした都市といった要素を抽象化」して設定を考える．

### ゲームジャンル

2D縦スクロール弾幕シューティングゲーム．

### プロジェクトの目標

C++/SDL2を利用して，2D弾幕シューティングゲームを開発し，Steamでの販売を目指す．ゲーム制作において次の要素に重点を置く：

- 大量の弾生成・管理
- オブジェクト指向+イベント駆動+コンポーネント指向(一部)による効率的な開発
- リプレイ機能・スコア管理といった「記録」要素
- 難易度調整の効率化
- コントローラへの対応
- Linux/Windows環境への対応

## ゲームコンセプト

### コアコンセプト

1プレイが短く，かつリトライ性の高い弾幕シューティングを目指す．プレイヤーは敵や敵弾を避けながら，ショット・ボムを使用して各ステージのボスを撃破して全ステージクリアを目指す．

ただし，ショットを常に撃ち続けるようなシューティングではなく，要所要所で弾を撃って，敵の攻撃を「突破」するシステムで，一般的な弾幕シューティングとの差別化を図る．

#### 常時撃たないとは

通常ショットは弾数制限があり，連射できないという制約を課す想定である．プレイヤーは何度も弾を撃って有利なフィールドを作るのではなく，弾を的確に撃って突破口を作る必要がある．

無駄撃ちは不利を招くし，「弾の装填」が必要であるから先を予測する必要がある．調整は難しくなるが，大きな差別要素になるはずである．

### ゲーム開発における要点

- 高い可読性
  - 第三者が読みやすい，解析しやすいことを常に意識する
- 難易度調整
  - イージー/ノーマル/ハードの3種類を想定
  - 各難易度ごとにマニュアルで調整するのではなく，各難易度に応じてパラメータ指定で難易度を調整できる
    - パラメータによる難易度の全体的な調整を管理する
    - 最終的には，プレイして視認性・回避可能性を確認し保証する必要がある
- 操作性が良好
  - 低遅延，細かい入力を適切に解釈できる
- パターン化
  - ランダムな弾幕生成だけでなく，弾幕のパターン化によるリプレイ性の向上
- 弾幕の視認性
  - このゲームにおいては，難易度の要素に「視認性の悪い弾」を利用しない
- プレイ時間が短い
  - 短時間プレイで何度も遊んでもらう
- スコアシステム
  - 過去のデータ/スコアを記録してリプレイ性を高める

### ゲームのボリューム

1ステージ3分，全3ステージで計10分程度の構成を最初は目指す．
ボリュームはスケジュールに応じて調整するが，最終的には6ステージ+EXステージの実装を目指す．

### ターゲットプレイヤ

- 2D縦スクロールシューティングゲームに対して
  - 何らかの作品に触れたことがある
  - 興味があって動画を見たことがある
  - 手軽にプレイできるゲームを求めている

上記のような弾幕STG初心者〜中級者を主対象とする．
難易度ノーマルが基本となり，初見では被弾しつつも数回のリトライで突破できる程度の難易度で設計する．
イージーは弾幕STGに不慣れなプレイヤー向けに，弾速と弾数を抑える．

## プロジェクトスコープ

重要な点として，4段階のスコープでプロジェクトを進める．

### 第零段階

ゲームの全体的な設計・技術的な検証を行う：

- 自機描画・移動
- 8方向移動・低速移動
- 自機弾・敵弾の実装・描画
  - 1000発の弾表示(性能テスト)
- 当たり判定の実装

  - 円の近似(矩形)の当たり判定
- デバッグ機能の実装
- 入力・ゲーム環境の記録処理
  - あくまで記録

この段階では，次のステップへ進むための明確な条件を設ける：

- 弾が1000発描画されても60FPSで動く
  - 具体例：Ubuntu 24.04 LTS の開発環境で，オブジェクトを表示・更新して平均60FPSを維持
    - 敵弾1000発のactive
    - 自機弾100発
    - 敵10体
- デバッグ情報を取得して次の情報を確認できる
  - frame time
  - update time
  - render time
  - active bullet count
- 自機が動ける・避けられる
  - 8方向の移動
  - 低速移動
- ゲーム中の各情報が収集できる
  - 活性状態の弾
  - 乱数seed固定
  - 入力の記録
- 各種オブジェクトの生成・更新・描画が安定している

※安定とは，1フレームの平均デルタ時間が1/60秒を超えてしまい，更新待ちが累積して**update処理が遅延するようなことが無い**ことを言う

## 技術的な要素

このセクションでは，2D縦スクロールシューティングゲームを作成するにあたって，技術的に重要な要素を考察する．

### 弾管理

大量の弾を都度生成/削除するのはパフォーマンスが悪いことは容易に想像がつく．弾はactive/inactiveによるbool値の状態管理で画面に表示する方針が良い．

- 弾の種類ごとに最大数を持つ`vector<Bullet>`を使う
- activeというフラグによる生存管理
- 弾の生成とは，inactive→activeへの状態遷移
- 画面外に消えた弾は即座にinactiveにする
- 弾道の計算はシンプルな計算式
  - 基本的なベクトル*パラメータの計算を行う
  - 複雑な制御はボスなどに限定する
- 最大弾数・使用率・データ量などは可能な限りデバッグ時に情報として表示させる

### 決定性

リプレイ機能を搭載する，テストを実施することを考えると，ゲームは再現可能性をなるべく持たせるのが望ましい：

- 固定タイムステップ
- フレーム単位の入力の受付
- 乱数を使用する場合はシード値を使う
- リプレイ機能とは，記録された入力をストリームで入力していくこと

### データ駆動

敵の出現・弾幕パターン・ボス戦をデータ駆動で定義する．
パターンは定義したパターン用の関数を呼び出すイメージ：

```json
{
    "time": 3.0, 
    "action": "spawn_enemy", 
    "enemy": "small_fairy", 
    "x": 320, 
    "y": -32, 
    "pattern": "spread_5way"
}
```

`"pattern": "spread_5way"`の部分が関数を指定している部分になる．

### コンポーネント指向(ECS)

弾は複数種類あるが，どれも似たような処理を適用する．また，大量に画面へ表示させる必要があることから，既存のエンジンにコンポーネント指向を組み合わせて，STG要素はコンポーネント指向で作っていく．

本プロジェクトでは，全てをコンポーネント指向にするのではなく，あくまでも大量に生成する必要があるオブジェクト(弾・アイテム・敵・エフェクトなど)に限定した，データ駆動ベースの配列管理を導入する．
連続メモリ上に配列として配置して，一括更新によるパフォーマンス向上を図る．
またデータ管理は原則データ駆動であり，シーンやボスなどはオブジェクト指向的管理になる想定である．

現状はArray of Structures(AoS)をベースとする．パフォーマンスを計測し，更新・当たり判定等の計算処理で期待したパフォーマンスが出ない場合に，Structure of Arrays(SoA)を検討する．

例：

- 弾(Bullet) + 自機狙い(Aimed)
- 弾(Bullet) + 複数方向(Multi-way) + レーザー(laser)

### 当たり判定

大量のオブジェクトが存在する都合上，当たり判定はシンプルかつ高速なものでなければならない．

- 連続メモリの配列
- 単純な四則演算による座標計算
- バッチ処理
  - 描画処理・当たり判定処理は極力まとめてオーバーヘッドの削減
- プレイヤーのヒットボックスは小さな円
- 当たり判定の形状は円 or 矩形
  - 円 vs 円
  - 円 vs 矩形
  - 矩形 vs 矩形
- 判定は原則線形全探索
  - 画面に存在するオブジェクト数に応じたグリッド分割
    - activeな弾・衝突判定数を計測し，パフォーマンス低下を回避するためにグリッド分割をするイメージ

### リプレイ機能

STGとして再現可能性を重要視する都合上，リプレイ機能は重要な用途と考える．しかし非常に難易度が高いと予想される．

リプレイは動画記録ではなく，入力記録でありゲームの再現であり，リプレイ開始時は，乱数は同じseedを使用するなど制限できる形式でゲームを作成していく必要がある．

初期実装としては，乱数seedの保存・入力記録の保存を目指す．ステージの開始〜終了ができた段階で，記録を流し入れて「再現」できることを確認する二段階で進める．

最終的にはファイル保存・読み込み可能な形式で出力できることを目指す．

### コントローラ対応

STGをプレイする上で，コントローラへの対応は必須に近いが実装は別問題である．まずはキーボードで操作できることを確認しなければならない．8方向の移動がスムーズにできるかなど，入力処理が適切に行われているかを確認した後，コントローラへの段階的な対応を始める．

基本的に，`SDL_GameController`のAPIを利用したゲームパッド対応を予定している．

### Windows対応

Steamで販売するにあたって，これは必須に近い．ただし，Windowsでの開発を並行するのは時間がかかる．基本的にはSDLとC++を用いてゲーム作成を行い，Windowsでコンパイルして動くゲームを作る，という方針で進めたい．

Windows対応は要調査である．

## 設計方針

The Mysterious Forestの資産を活かし，メイン処理，サブシステム，一部新システムへコンポーネントを組み込むハイブリッド方式で進める．

| Area | Responsibility | Note |
| :---: | :---: | :---: |
| Game | メインループ・シーン管理 |  |
| SceneManager | シーン遷移の管理 | Title->Playing->... |
| ResourceManager | リソースの管理 | 前作のようにGameクラスで管理せず切り分ける |
| InputSystem | 入力受付 | 既存 |
| PlayerSystem | プレイヤーの更新 |  |
| BossSystem | ボスの更新 |  |
| EnemySystem | ID+Systemによるコンポーネント化 | 連続したメモリ配置でパフォーマンス向上 |
| BulletSystem | ID+Component+Systemによるコンポーネント化 | 弾を大量に表示する都合上コンポーネント化は必須 |
| ItemSystem | ID+Componentによるコンポーネント化 |  |
| StageScriptSystem | 時刻に応じた敵出現・イベント発火 | ステージの設計書の読み込みを担当 |
| PatternSystem | 弾幕パターン生成 | ステージの設計とは別 |
| CollisionSystem | 各種オブジェクトの当たり判定 |  |
| DebugOverlay | FPS/弾数/処理時間表示 | デバッグ情報は各クラスファイルに仕込むのではなく，専用のクラスで実装する |
| AudioSystem | SE/BGM |  |
| ReplaySystem | リプレイ用の記録・再生 | フレーム開始時点でデータを収集するイメージ |
| SaveSystem | スコア・設定等管理用 | 基本的にはJsonベース |

### ルール

画面に複数個登場するようなオブジェクト・エンティティはコンポーネント指向によるパフォーマンス向上を目指す．
特定のクラスが肥大化することは避け，システムの相互呼び出しによって1つのシステムが肥大化しないようにする．
